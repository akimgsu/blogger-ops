2. Specificity and the Cascade

As the name Cascading Style Sheets (CSS) implies, styles flow downward like a waterfall, cascading over elements. However, when multiple CSS rules target the exact same element, the browser relies on a strict scoring system called Specificity to determine which styles win.

How to Calculate Specificity Weight

Think of specificity as a four-tier point system: (Inline, ID, Class, Tag)

  1. Inline Styles (1, 0, 0, 0): Styles applied directly within the HTML element via the style="" attribute. These carry the heaviest weight.
  2. ID Selectors (0, 1, 0, 0): Selectors like #header.
  3. Class, Pseudo-class, Attribute Selectors (0, 0, 1, 0): Selectors like .btn, :hover, or [type="text"].
  4. Tag, Pseudo-element Selectors (0, 0, 0, 1): Selectors like div, p, or ::before.

For example, .card .title has two classes, yielding a score of (0, 0, 2, 0). Meanwhile, #title has one ID, yielding (0, 1, 0, 0). The higher weight always wins. What if the scores are tied? The rule of the Cascade takes over: the style declared last (closest to the bottom of the stylesheet) overrides the previous ones.

The Temptation and Danger of !important

!important is a cheat code. It forcefully bypasses the entire specificity point system.

.btn {
  background-color: blue !important;
}

When styles refuse to apply, beginners often fall into the trap of sprinkling !important everywhere. However, abusing this completely breaks the natural cascade of CSS. It inevitably leads to a maintenance nightmare where you have to write even more !important declarations just to override previous ones.

The Best Practice for Resolving Conflicts:
If a style isn't applying, don't reach for !important. Open your browser's Developer Tools to inspect which selector is outscoring yours. The correct approach is to refactor your code or organically increase your selector's weight by chaining an additional class (e.g., adding a parent class context).
